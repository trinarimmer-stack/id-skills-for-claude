# Contributing to ID Skills for Claude

Thank you for wanting to make this library better. 
Contributions from working instructional designers 
are what will keep this resource sharp, current, 
and genuinely useful.

---

## What We're Looking For

### New skill files
Coverage of an instructional design domain 
not yet in the library. Before writing, 
check the existing files and open an Issue 
to propose the topic — someone may already 
be working on it.

### Improvements to existing files
- Frameworks or models that belong 
  in an existing skill but are missing
- Outdated references or superseded research
- Sections that are unclear, incomplete, 
  or could be more practical
- Additional prompt examples or checklists

### Bug fixes
- Broken formatting
- Inaccurate citations or references
- Factual errors

---

## Standards for Skill Files

Every skill file in this library is held 
to the same bar. Before submitting, 
check your file against these criteria:

### Structure
- [ ] Follows the skill file template 
      (`SKILL_TEMPLATE.md`)
- [ ] Filename uses snake_case with 
      `.md` extension
- [ ] Filed in the correct folder 
      (`foundations/`, `design_practice/`, 
      `craft_and_quality/`, or 
      `collaboration_and_strategy/`)
- [ ] One skill per file — 
      no omnibus files covering 
      multiple unrelated topics

### Content
- [ ] Opens with a clear Overview 
      that explains what the skill covers 
      and why it matters
- [ ] Includes a Core Principle — 
      the most important idea in the domain
- [ ] Covers established, evidence-informed 
      frameworks and practices
- [ ] Includes at least one practical 
      checklist or set of actionable prompts
- [ ] Closes with a References section 
      with credible, practitioner-accessible sources
- [ ] Does not duplicate content 
      already covered in another file

### Voice and Tone
- [ ] Written for working instructional designers — 
      practical, not academic
- [ ] Opinionated where appropriate — 
      takes stances, not just presents options
- [ ] Plain language — no jargon 
      without definition
- [ ] Consistent with the tone 
      of existing files in the library

### Quality
- [ ] Proofread — no spelling or 
      grammatical errors
- [ ] Formatting consistent with 
      existing files
- [ ] All claims supported by 
      credible sources

## How to Contribute

1. **Fork the repository**

2. **Create a branch**
   Name it descriptively:
   `add-learning-experience-design`
   `update-assessment-design-feedback-section`

3. **Write or edit your file**
   Use `SKILL_TEMPLATE.md` as your starting point 
   for new files. Follow the standards above.

4. **Test it**
   Load your skill file into Claude and 
   try at least three prompts. Does Claude 
   respond with genuinely useful, 
   instructionally sound output? 
   If not, the file needs more work.

5. **Submit a pull request**
   Include:
   - What the file covers and why it belongs 
     in the library
   - Which folder it lives in and why
   - Any sources or frameworks you drew from
   - Confirmation that you've tested it with Claude

6. **Respond to review feedback**
   Pull requests may receive feedback 
   before merging. This is normal and 
   makes the library better.

## Folder Guide

| Folder | What belongs here |
|---|---|
| `foundations/` | Core ID knowledge and process — needs analysis, objectives, writing, cognitive load, ID models |
| `design_practice/` | How to design specific types of learning experiences — scenarios, interactions, motivation, behavior change, skills development |
| `craft_and_quality/` | How to execute and evaluate design work — visual design, accessibility, QA |
| `collaboration_and_strategy/` | How to work with others and use tools effectively — stakeholder management, AI prompting |

If your skill doesn't clearly fit an existing folder, 
open an Issue to discuss before filing it.

---

## Code of Conduct

This library is a professional resource 
built by and for instructional designers. 
Contributors are expected to:

- Be respectful and constructive in 
  all interactions
- Give credit to sources and frameworks
- Prioritize accuracy and usefulness 
  over novelty
- Welcome feedback as a quality signal, 
  not a personal critique

---

## Questions?

Open an Issue and tag it `question`. 
We'd rather answer a question upfront 
than review a file that misses the mark.
